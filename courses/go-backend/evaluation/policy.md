# Go 日课评测政策

## 权威来源

只使用指定 Day N 的课程文件和学习者提交的当天证据。课程中的以下栏目共同定义评测范围：

- 学习目标。
- Node.js 对照。
- Go 核心心智。
- 实践步骤与建议产物。
- 测试/验证命令。
- 检索问题。
- 常见误区。

不得使用其他 Day、模型常识或外部资料提高当天标准。

## 动态评分

先从“学习目标”提取能力项，再用其他栏目补充该能力项的证据要求，避免把同一知识重复计分。只有课程明确要求的产物和验证才成为独立能力项。

每项使用五级诊断分：

| 等级 | 判定 |
|---:|---|
| 4 | 准确解释机制、边界与当天要求的取舍，并有充分证据 |
| 3 | 准确、完整地满足当天要求 |
| 2 | 部分正确，但缺少机制、边界或必要证据 |
| 1 | 只有术语、结论或未经验证的主张 |
| 0 | 错误、缺失或与当天课程冲突 |

参考分数按 `已得等级之和 / (能力项数量 × 4) × 100` 计算并四舍五入。只有每个必修能力项都达到 3 级或以上，最终状态才是“通过”。

评分目的是定位缺口，不排名、不设置奖励，也不允许高分抵消单项不合格。

## 问答纪律

1. 每次只问一道题。
2. 首次优先使用课程原有检索问题。
3. 仅在回答含糊或回炉重评时受限改写。
4. 改写必须评测同一能力项，只能使用当日课程已有术语。
5. 不固定题目数量；覆盖全部必修能力项后才能结束。
6. 用户回答必须写入 `notes.md`，评测器不得代写或润色。

## 定向回炉

未达标时只记录：

- 未达标能力项。
- 问题类型：概念混淆、机制缺失、边界不清或证据不足。
- 判定依据在用户回答或产物中的位置。
- 需要重读的当日课程小节标题。
- 当前尝试次数。

随后要求用户重读该小节并重新回答同一能力项。每项最多三次；第三次仍低于 3 级时，标记“重新学习”并停止当天评测。

## 零答案泄露

严禁输出：

- 标准答案、参考实现、正确代码或修改后的完整表述。
- 正确 API、关键术语、伪代码或推导步骤。
- 能暗示正确方向的反问、选项排除或相邻知识补充。
- 其他 Day 的内容。

允许输出失败标准、问题类别、证据位置和课程小节标题。引用用户回答时只引用定位问题所需的最短片段，并先排除敏感信息。

## 工程证据

只有当日课程明确要求时，才检查代码、测试、migration、proto、阅读笔记或命令结果。引用路径必须在当前仓库内，并由 `notes.md` 明确列出。

允许自动运行的命令仅限以下前缀：

```text
go test
go test -race
go vet
go test -bench
```

命令必须直接执行，不得包含 `;`、`&&`、`||`、管道、重定向、反引号、`$()`、环境变量赋值或网络地址。可能访问数据库、网络或外部服务的测试不得自动运行。

生成、格式化、migration 和数据库写入只检查学习者证据，不由评测器执行。

## 评测文件

`learning-records/go-backend/lessons/<lessonId>/evaluation.md` 是唯一 Evaluation 事实源，只能由共享 Evaluation Core 原子更新。当前状态、0–4 能力等级、尝试次数与追加事件必须保持可校验一致；通过不可回退，旧 evaluationRevision 历史不得删除。

不得在 Evaluation 中保存标准答案、答案提示、密钥、DSN 或完整隐私数据。Notes、Exercise、Release Progress Snapshot、Public Progress 和 Roadmap 均不得反向改写 Evaluation。
